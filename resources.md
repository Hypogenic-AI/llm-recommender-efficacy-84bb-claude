# Resources Catalog

## Summary

Resources gathered for testing the hypothesis: *"If a large language model is
given exported Spotify data and asked for music recommendations, its
recommendations may be better than those generated by Spotify's own
recommendation algorithms."*

19 papers, 2 datasets (114K-track Spotify catalog + 50-user Last.fm listening
history sample, plus pre-generated NL profiles bundled in one of the cloned
repos), and 6 code repositories were collected.

## Resource gathering notes

### Search strategy
1. Ran paper-finder in diligent mode on "large language models for music
   recommendation systems" → 302 papers, 21 at relevance=3.
2. Ran paper-finder in fast mode on "zero-shot LLM recommender system
   evaluation" → 74 papers, 41 at relevance=3.
3. Cross-referenced citation-heavy LLM-as-recommender papers from the search
   results with web search to confirm arxiv IDs and locate code repos.
4. Selected ~20 most relevant papers spanning foundational (2023, high
   citation), music-specific (2025), and methodology/evaluation (2025).

### Selection criteria
- Foundational papers with >150 citations from 2023 (Hou, Dai, Sanner, He,
  Kang, Wang & Lim, Lin survey).
- All recent music-specific LLM recommendation papers (Sguerra, Boadana,
  TalkPlay family, Text2Tracks, Yun, JAM, Epure survey).
- One head-to-head benchmark spanning music (RecBench).

### Challenges encountered
- Spotify's official Million Playlist Dataset is no longer publicly
  downloadable; used Last.fm 1K + Spotify Tracks Dataset as a substitute
  pairing.
- The most relevant paper (Boadana et al. 2025) is in Portuguese; results
  table and methodology are still extractable from the figures and tables.
- Paper-finder failed initially due to missing `httpx` dependency; resolved by
  `uv add httpx`.

### Gaps and workarounds
- We could not find a single dataset that pairs Spotify export listening
  history with a queryable Spotify catalog. Workaround: Last.fm-as-history +
  Spotify-Tracks-as-catalog, joined on (artist_name, track_name) when needed.
- Spotify production algorithm is not directly comparable. Use multiple
  classical baselines (EASE, item-kNN, CBF) as proxies and frame results in
  terms of "LLM vs. each classical baseline" rather than "LLM vs. Spotify".

## Recommendations for experiment design

(Mirrors the closing section of `literature_review.md`.)

1. **Primary dataset**: Last.fm 1K (50-user sample to start, 1K for power) as
   listening history; HuggingFace Spotify Tracks Dataset as item catalog.
2. **Baselines**: EASE (closed-form CF), item-kNN, TF-IDF/cosine on genre
   tags (Boadana CBF analogue), popularity-only sanity baseline.
3. **LLM arms**: zero-shot ranking with sequential prompt (Hou template);
   NL-profile + bi-encoder retrieval (Sguerra template). Optionally a
   tool-using agent (Boadana / TalkPlay-Tools template).
4. **Metrics**: Recall@10, NDCG@10, ARP, LongTailShare,
   EntitiesResolvedShare (hallucination), Genre Diversity@10, Personalization
   Gain. Optionally Like Rate / Playlist Rating if a user study is run.
5. **Methodological musts**:
   - Hold out last 30 days per user as test set.
   - Bootstrap candidate orderings for LLM ranker (position bias).
   - Match LLM output to catalog; report hallucination rate.
   - Log token cost and latency.
   - Stratify by genre and popularity tier.
   - Report ≥3 runs at T=0; show robustness at T=0.7.
